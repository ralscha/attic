portal-java 0.7 demos

Demos

  • Chat

    This is the sample application from portal-java.

  • Bryntum Ext Scheduler / Bryntum Touch Scheduler

    This is a port from a Ext JS/Sencha Touch/Node.js sample application published in this blog post: http://www.bryntum.com/blog/under-the-hood-of-the-socket-io-express-example/.

  • Twitter

    Twitter demo that uses the Streaming API from Twitter and tracks these terms "ExtJS", "Sencha", "atmo_framework", "#java", "java7", "java8", "websocket", "#portal", "html5" and "javascript".

  • Snake

    Translation of the Websocket snake demo included in the Tomcat 7 distribution (webapps/examples folder).

  • Ext JS Grid

    Demonstrates an Ext JS store and model with a portal.js proxy. Create, update and delete operations are broadcasted to all connected clients.

  • Access Log Wargames Map

    Translates ip addresses from the Apache access log from this server to latitude/longitude with the help of GeoCityIP and visualizes them on a world map. This example is based on maptail and mape's wargames.

  • Smoothie Charts

    Copy of the Smoothie Charts Server Load Example. But in this example the server generates the random data every second and sends it to the connected clients.

  • Google Maps Demo

    Server periodically sends coordinates to the connected clients that moves two car icons on a google maps.

  • Chat with WebRTC

    An enhanced chat application with WebRTC. The public chat should work in any browser and is implemented like the simple chat example above.
    The WebRTC part only works in Chrome. Your computer also needs a webcam. Test it with a friend that also has Chrome or test it from a single computer by opening two tabs. Connect to the chat with different names, then one user selects the other user and clicks the button "Start Peer-to-Peer Connection". If everything goes right you should see the video from the other user/tab on the right.
    The SDP and ICE messages between the two peers are exchanged with portal. As soon as the connection is established it's a peer to peer connection that no longer needs a server. For more information about WebRTC see the article on html5rocks.
    The peer-to-peer part should also work over the internet even if both users are behind a NAT/Firewall. The application uses a STUN server (stun.stunprotocol.org) for external ip and port discovery. But it may not work everywhere. When NAT 'hole punching' fails it needs a TURN server to relay the traffic, but I haven't found a free and public available TURN server that I could use for this demo.
